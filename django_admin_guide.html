üìå Introduction

Django's admin panel is one of its most powerful features, allowing you to manage your application data with minimal code. By default, the admin interface is functional but basic. This guide shows you how to customize it to create an intuitive, professional admin experience tailored to your needs.

What You'll Learn: How to reorganize fields, control display options, manage related models, customize list views, and make your admin interface more user-friendly.

üöÄ Basic Setup: Creating a ModelAdmin Class

To customize the admin interface, you need to create a ModelAdmin class and register it with Django's admin site.

Step 1: Basic Registration

Here's the standard way to register a model:

from django.contrib import admin from .models import Question admin.site.register(Question)
# Simple registration (no customization)
Step 2: Create a ModelAdmin Class

For customization, replace the simple registration with a ModelAdmin class:

from django.contrib import admin from .models import Question admin.site.register(Question)
üí° Tip:
Now you can add customization options to the QuestionAdmin class as shown in the sections below.

üìã 1. Controlling Field Order

By default, Django displays fields in the order they're defined in your model. Use the fields attribute to customize the order on the edit form.

class QuestionAdmin(admin.ModelAdmin): fields = ["pub_date", "question_text"] # Custom field order admin.site.register(Question, QuestionAdmin)
Result: The fields "pub_date" and "question_text" will appear in that specific order on the edit form, regardless of their order in the model.

üóÇÔ∏è 2. Organizing Fields with Fieldsets

Fieldsets group related fields under collapsible sections, making forms more organized and easier to navigate.

class QuestionAdmin(admin.ModelAdmin): fieldsets = [ ('Question', { 'fields': ['question_text'] }), ('Date Information', { 'fields': ['pub_date'], 'classes': ['collapse'] # Optional: makes section collapsible }), ] admin.site.register(Question, QuestionAdmin)
How It Works: Each fieldset is a tuple containing a title and a dictionary with field options. The 'collapse' class makes the section collapsible (useful for advanced options).
‚ú® Advanced Fieldset Options:
  • classes=['collapse'] - Makes the section collapsible
  • classes=['wide'] - Makes input fields wider
  • description - Add explanatory text to a fieldset
fieldsets = [ ('Question Details', { 'fields': ['question_text'], 'description': 'Enter the question that will be displayed to users.' }), ('Advanced Options', { 'fields': ['pub_date'], 'classes': ['collapse', 'wide'], 'description': 'Set the publication date and other metadata.' }), ]

üîó 3. Managing Related Models with Inline

When your models have relationships (ForeignKey, OneToOneField), you can edit related objects directly within the parent model's form using Inline classes.

Two Types of Inline:
Type Display Style Best For
StackedInline Vertical layout (one item per row) Complex forms with many fields per related object
TabularInline Table format (compact) Simple forms with few fields per related object
Using StackedInline (Vertical Layout)
from django.contrib import admin from .models import Question, Choice class ChoiceInline(admin.StackedInline): model = Choice extra = 0 # Don't show empty extra forms min_num = 1 # Require at least 1 choice max_num = 10 # Allow maximum 10 choices class QuestionAdmin(admin.ModelAdmin): fields = ['pub_date', 'question_text'] inlines = [ChoiceInline] admin.site.register(Question, QuestionAdmin)
Using TabularInline (Table Format)
class ChoiceInline(admin.TabularInline): model = Choice extra = 0 min_num = 1 max_num = 10 class QuestionAdmin(admin.ModelAdmin): inlines = [ChoiceInline] admin.site.register(Question, QuestionAdmin)
‚öôÔ∏è Inline Configuration Options:
  • extra - Number of empty forms to display (default: 3)
  • min_num - Minimum number of forms required
  • max_num - Maximum number of forms allowed
  • readonly_fields - Fields that can't be edited
  • fields - Control which fields are displayed

üìä 4. Customizing the List View

The list view displays all objects in a table format. Customize it to show relevant information at a glance.

Show Specific Columns

Use list_display to control which fields appear in the list view:

class QuestionAdmin(admin.ModelAdmin): list_display = ['question_text', 'pub_date', 'get_choice_count'] def get_choice_count(self, obj): return obj.choice_set.count() get_choice_count.short_description = 'Number of Choices' admin.site.register(Question, QuestionAdmin)
Make Fields Clickable And Filter Sidebar

Control which fields link to the detail page using list_display_links and filter sidebar using list_filter:

class QuestionAdmin(admin.ModelAdmin): list_display = ['question_text', 'pub_date'] list_display_links = ['question_text','pub_date'] # Make question_text and pub_date are clickable list_filter = ["pub_date"] # Add a filter sidebar by pub_date admin.site.register(Question, QuestionAdmin)
Edit Directly from List View

Allow users to edit certain fields without opening the detail page using list_editable:

class QuestionAdmin(admin.ModelAdmin): list_display = ['question_text', 'pub_date'] list_editable = ['pub_date'] # Can edit pub_date directly in list admin.site.register(Question, QuestionAdmin)
Note: Fields in list_editable cannot be in list_display_links. At least one field must be a link.

‚öôÔ∏è 5. Advanced List View Customization

Default Sort Order

Set the default sorting with ordering:

class QuestionAdmin(admin.ModelAdmin): ordering = ['-pub_date'] # Sort by pub_date descending (newest first) admin.site.register(Question, QuestionAdmin)
Use '-' prefix to sort in descending order. Multiple fields are supported: ordering = ['-pub_date', 'question_text']
Pagination

Control how many items appear per page using list_per_page:

class QuestionAdmin(admin.ModelAdmin): list_per_page = 10 # Show 10 items per page (default: 100) admin.site.register(Question, QuestionAdmin)
Date-Based Navigation

Add date hierarchy for quick filtering using date_hierarchy:

class QuestionAdmin(admin.ModelAdmin): date_hierarchy = 'pub_date' # Add date filter at top admin.site.register(Question, QuestionAdmin)
üìÖ How Date Hierarchy Works:
  • Shows year links at the top
  • Click a year to see months
  • Click a month to see days
  • Click a day to filter by that specific date

üîç 6. Adding Search and Filters

Add Search Box

Enable searching with search_fields:

class QuestionAdmin(admin.ModelAdmin): search_fields = ['question_text', 'pub_date'] # Search across these fields admin.site.register(Question, QuestionAdmin)
Add Filters

Add filter sidebar with list_filter:

class QuestionAdmin(admin.ModelAdmin): list_filter = ['pub_date'] # Filter by publication date search_fields = ['question_text'] admin.site.register(Question, QuestionAdmin)
Supported Filter Types: Model fields (CharField, DateField, BooleanField, ForeignKey) automatically generate appropriate filters.

üéØ Complete Example: Fully Customized Admin

Here's a comprehensive example combining all customization options:

from django.contrib import admin from .models import Question, Choice class ChoiceInline(admin.TabularInline): model = Choice extra = 0 min_num = 1 max_num = 10 class QuestionAdmin(admin.ModelAdmin): # Field organization fieldsets = [ ('Question Details', { 'fields': ['question_text'], 'description': 'Enter the question text here.' }), ('Publishing', { 'fields': ['pub_date'], 'classes': ['collapse'], }), ] # Related models inlines = [ChoiceInline] # List view customization list_display = ['question_text', 'pub_date', 'get_choice_count'] list_display_links = ['question_text'] list_filter = ['pub_date'] list_per_page = 20 date_hierarchy = 'pub_date' # Search search_fields = ['question_text'] # Ordering ordering = ['-pub_date'] # Custom method for list display def get_choice_count(self, obj): return obj.choice_set.count() get_choice_count.short_description = 'Total Choices' admin.site.register(Question, QuestionAdmin)

‚úÖ Best Practices

1. Keep It User-Friendly

Only display fields that users need to see or edit. Too many columns make the interface confusing.

2. Use Descriptive Names

Add short_description to custom methods for clarity in the admin interface.

3. Organize with Fieldsets

Group related fields in fieldsets to improve form usability for large models.

4. Choose the Right Inline Type

Use TabularInline for simple relationships and StackedInline for complex ones.

5. Limit List Display Columns

Show only 3-5 relevant columns to avoid horizontal scrolling on smaller screens.

6. Enable Appropriate Filters

Add filters for fields that users will frequently search or filter by.
